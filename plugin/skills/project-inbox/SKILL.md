# Project Inbox

## Purpose

Turn the passive `documents/inbox/` holding area into an intelligent staging ground. Instead of waiting for `project-document-curator` to manually classify each file, this skill reads every unprocessed document, extracts meaning, assigns a use designation, and builds a structured orientation summary that the onboarding skill can use to pre-populate context before the user answers a single question.

The core payoff: drop files, run `/project-inbox`, then run `/project-onboarding` — and onboarding becomes a confirmation pass instead of a blank-slate interview.

---

## Sub-actions

### `triage` (default)

Run the full classification pass on all documents with `triage_state: unprocessed` in `documents/index.yaml`, plus any files in `documents/inbox/` not yet indexed.

**For each document:**

1. **Read the document.** For text-based files (md, txt, csv): read the full content. For binary (pdf, docx, xlsx, png): read filename, extension, any available metadata, and attempt to extract text if tools permit. For images: describe visible content.

2. **Classify it** across these dimensions:

   **Doc type** — use the same 13-type enum as `project-document-curator`:
   - `mpa`, `mpa-template`, `schedule-a-workbook`, `schedule-a-template`, `proposal`, `approval-letter`, `financial-assessment`, `pic-template`, `capital-cost-request`, `change-order`, `claim-submission`, `sc-agenda`, `sc-minutes`, `monthly-brief`, `weekly-report`, `publication`, `presentation`, `ip-disclosure`, `annual-questionnaire`, `final-report`, `invoice`, `misc`

   **Use designation:**
   - `imprint` — shapes project identity; useful for orienting project-state before or during onboarding (governing documents, proposals, SOWs, milestone schedules, org charts, project briefs)
   - `for-use` — a working document the team will actively use (a template to fill, a form to submit, a reference sheet)
   - `example` — a sample output or past work the team wants to model (previous reports, claim examples, presentations that landed)
   - `output` — an artifact generated by project-state (auto-detect by filename pattern: reports/, claims/, meeting packs)
   - `unknown` — cannot determine without more context

   **Relevance score** (0–100): How useful is this document for orienting project-state?
   - 80–100: Governing documents, proposals, signed agreements, milestone schedules
   - 50–79: Reports, meeting minutes, financial workbooks, stakeholder lists
   - 20–49: Templates, blank forms, generic reference material
   - 0–19: Output artifacts, irrelevant, or empty

   **Action flags** (zero or more):
   - `seed-manifest` — contains project name, funder, dates, or consortium info
   - `seed-milestones` — contains a milestone list, schedule, or work breakdown
   - `seed-risks` — contains a risk register or risk narrative
   - `seed-people` — contains a stakeholder list, org chart, or named contacts
   - `seed-budget` — contains budget figures or financial breakdown
   - `example-output` — suitable for `references/examples/good/`
   - `reference-material` — general reference; useful to keep available

   **Extraction summary**: 2–3 sentences describing the document's content and its value to the project substrate. Be specific: name milestone counts, funder names, dates, and key facts if found.

   **Triage confidence**: `high` | `medium` | `low` based on document readability and classification certainty.

3. **Write enriched metadata** to `documents/index.yaml`. For documents already indexed: extend in place, preserving all existing fields. For documents not yet indexed: create a new entry. New fields added:

   ```yaml
   triage_state: processed
   use_designation: imprint
   relevance_score: 88
   action_flags:
     - seed-manifest
     - seed-milestones
   extraction_summary: "Signed MPA covering five milestones across three consortium partners, total project value $2.1M CAD, project window 2024-01-01 to 2026-06-30. Primary governing document — seed manifest and milestone list from this."
   triage_timestamp: "YYYY-MM-DDTHH:MM:SSZ"
   triage_confidence: high
   ```

4. **Route imprint documents.** For any document where `use_designation: imprint`:
   - Copy (not move) the file to `references/imprint/<doc-id>-<slug>.<ext>`
   - Append an entry to `references/imprint/index.yaml`:
     ```yaml
     - id: doc-id
       path: references/imprint/doc-id-slug.pdf
       source_path: documents/inbox/original-filename.pdf
       kind: mpa
       action_flags: [seed-manifest, seed-milestones]
       relevant_chapters: [2, 4, 5]  # which onboarding chapters this doc pre-fills
       extraction_summary: "..."
       triage_timestamp: "..."
     ```
   - Map `action_flags` to `relevant_chapters`: `seed-manifest` → [2], `seed-milestones` → [5], `seed-people` → [4], `seed-budget` → [2], `seed-risks` → [7]

5. **Log** to `logs/activity.ndjson`:
   ```json
   {"event": "inbox.triage.document", "doc_id": "...", "designation": "imprint", "relevance_score": 88, "timestamp": "..."}
   ```

After processing all documents, run `orient` automatically to regenerate `references/inbox-orientation.yaml`.

**Output:** Present a triage summary table:
```
Inbox Triage Complete — N documents processed

ID                    Kind              Designation    Score   Flags
────────────────────────────────────────────────────────────────────
doc-001-project-brief proposal          imprint         88     seed-manifest, seed-milestones
doc-002-schedule-a    schedule-a-wb     imprint         92     seed-manifest, seed-milestones, seed-budget
doc-003-example-rpt   weekly-report     example         55     example-output
doc-004-blank-form    pic-template      for-use         22     reference-material

Imprint documents (2) copied to references/imprint/
Orientation generated → references/inbox-orientation.yaml
```

### `status`

Show triage state summary without making changes.

Report:
- Count by `triage_state`: processed / unprocessed / dismissed
- Count by `use_designation`: imprint / for-use / example / output / unknown
- Unindexed files (in `documents/inbox/` but not in `documents/index.yaml`)
- Whether `references/inbox-orientation.yaml` exists and its last generated timestamp
- Quick recommendation: "Run `project-inbox triage` to process N unprocessed documents"

---

### `flag <doc-id> <designation>`

Manually override a document's `use_designation`. Valid designations: `imprint`, `for-use`, `example`, `output`, `unknown`.

Steps:
1. Find the entry in `documents/index.yaml` by `id`.
2. Update `use_designation` to the provided value.
3. If changing to `imprint` and not already in `references/imprint/`: copy file and update `references/imprint/index.yaml`.
4. If changing away from `imprint` and in `references/imprint/`: note (do not delete the copy — leave it with a `manually_demoted: true` flag).
5. Log `inbox.designation.manual_override` to activity log.
6. Suggest running `orient` to refresh orientation: "Run `project-inbox orient` to update the orientation summary."

---

### `orient`

(Re)generate `references/inbox-orientation.yaml` from all documents currently flagged as `imprint`.

Read all entries in `references/imprint/index.yaml`. For each, read the associated extraction summary and action flags. Synthesize into a structured orientation summary:

```yaml
# Generated by project-inbox orient — YYYY-MM-DD
# This file is a synthesized summary of imprint-flagged documents.
# Used by project-onboarding Chapter 0 to pre-populate context.
# Do not edit manually — regenerate with /project-inbox orient

generated_at: "YYYY-MM-DDTHH:MM:SSZ"
document_count: 2
confidence: high  # high if all imprint docs are high-confidence; medium if mixed; low if any low

project_identity:
  short_name_hint: "ABC-XYZ"         # extracted if found, null if not
  long_name_hint: "Full Project Name"
  funder_hint: "Funder Name"
  program_hint: "Program Name"
  start_date_hint: "2024-01-01"
  end_date_hint: "2026-06-30"
  budget_hint_cad: 2100000
  governing_document_id: "doc-002-schedule-a"

stakeholder_hints:
  - name: "Jane Smith"
    organization: "OrgA"
    role_hint: "Project Lead"
    source_doc: "doc-001-project-brief"
  # add one entry per clearly named person found across imprint docs

milestone_hints:
  - title_hint: "M01 — Data Infrastructure Setup"
    owner_hint: "OrgA"
    end_date_hint: "2024-06-30"
    source_doc: "doc-002-schedule-a"
  # add one entry per milestone found

pre_filled_chapters:
  # which onboarding chapters can be presented as confirmations (not blank questions)
  "2": {confidence: high, fields: [short_name, long_name, funder, program, start_date, end_date, budget]}
  "4": {confidence: medium, fields: [stakeholders]}
  "5": {confidence: high, fields: [milestones]}

gaps:
  # fields that appeared in no imprint document
  - field: "project.one_liner"
    affects_chapters: [2]
  - field: "stakeholders.preferences"
    affects_chapters: [4]

imprint_sources:
  - doc_id: "doc-001-project-brief"
    path: "references/imprint/doc-001-project-brief.pdf"
    kind: "proposal"
    contribution: "Project name, funder, overview narrative, team names"
  - doc_id: "doc-002-schedule-a"
    path: "references/imprint/doc-002-schedule-a.xlsx"
    kind: "schedule-a-workbook"
    contribution: "Full milestone list with dates, owners, and budget allocation"
```

Log `inbox.orientation.generated` to activity log.

---

### `clear <doc-id>`

Mark a document as dismissed without full classification.

Steps:
1. Set `triage_state: dismissed` in `documents/index.yaml`.
2. If the document is in `references/imprint/`, warn: "This document has already been copied to references/imprint/ — clearing it won't remove that copy. Use `flag <id> unknown` if you want to demote it."
3. Log `inbox.document.dismissed` to activity log.

## documents/index.yaml schema extension

The following fields are added to existing document index entries by this skill. All are optional and backward-compatible — existing entries without them are treated as `triage_state: unprocessed`.

```yaml
entries:
  - id: doc-001
    path: documents/inbox/project-brief.pdf
    kind: proposal                          # existing
    status: inbox                           # existing
    source_of_truth: false                  # existing
    # --- smart inbox fields (added by project-inbox) ---
    triage_state: processed                 # processed | unprocessed | dismissed
    use_designation: imprint                # imprint | for-use | example | output | unknown
    relevance_score: 88                     # 0-100
    action_flags:                           # zero or more
      - seed-manifest
      - seed-milestones
    extraction_summary: "Two-page project brief covering consortium structure..."
    triage_timestamp: "2026-05-12T14:23:00Z"
    triage_confidence: high                 # high | medium | low
```

## references/imprint/ directory

Created on first `triage` run when imprint-flagged documents are found.

```
project-state/references/imprint/
├── index.yaml                         # index of all imprint documents with roles
├── doc-001-project-brief.pdf          # copy of original
└── doc-002-schedule-a.xlsx            # copy of original
```

`references/imprint/index.yaml` entries include `relevant_chapters` so `project-onboarding` Chapter 0 can route pre-fills precisely.

## Integration with project-onboarding

`project-onboarding` should check for `references/inbox-orientation.yaml` at startup (before Chapter 0). See `project-onboarding` Chapter 0 — Inbox Orientation for the full protocol.

---

## Integration with project-document-curator

When `project-document-curator` encounters a document that already has `triage_state: processed` in `documents/index.yaml`, it should:
- Use the smart inbox `use_designation` and `extraction_summary` as the starting classification hypothesis
- Present it to the user as a confirmation ("Smart inbox pre-classified this as `proposal` / `imprint` — confirm?") rather than running the 5-question process
- Preserve all smart inbox metadata fields when writing the updated entry

## Reads

- `documents/inbox/` — all files (unprocessed)
- `documents/index.yaml` — existing index (extend in place)

## Writes

- `documents/index.yaml` — adds/extends entries with triage metadata
- `references/imprint/<doc-id>-<slug>.<ext>` — copies of imprint documents
- `references/imprint/index.yaml` — imprint document registry
- `references/inbox-orientation.yaml` — structured orientation summary
- `logs/activity.ndjson` — triage events (append-only)

## Called by

- User directly
